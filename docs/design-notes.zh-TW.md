# 知識引擎設計筆記：非直覺的取捨

> 本文深入支撐整套系統的**知識引擎**，是 README 設計巧思（系統層）的底層展開，並附**示意**程式片段（皆為化名，非真實程式碼）。
> 這份文件的重點在於記錄當下的設計判斷——**為什麼這樣選**，而非操作手冊。

---

## 1. 為什麼不用 RAG：把「結構化」提前到 write-time

最直覺的做法是上 RAG（向量檢索）：清洗 → 切 chunk → 嵌入 → 向量庫 → 相似度召回。我在前身系統（一套處理長文本的 RAG 管線）上試過，搬到遊戲開發資料卻嚴重水土不服，歸納出三個病根：

| 病根 | RAG 的問題 | 本系統的解法 |
|------|-----------|-------------|
| **結構被碾平** | 結構化的規格被切成數百字的 chunk 嵌成一團，問不出「某次改版到底動了哪幾條設定」 | LLM 在**策展時**保留結構，寫成可讀的 wiki 主題頁 |
| **定案分不出** | 討論串切塊後，被推翻的中間意見與最終定案語意相近，LLM 分不清誰才是權威 | 知識頁帶明確 `status`（討論中 / 定案），定案是被標記的權威記錄 |
| **來源 / 決策靠猜** | 「誰決定的、資源在哪」只能靠向量相似度撈，不可靠 | 策展時把來源 / 時間 / 權威寫成 Frontmatter 結構欄位，查詢時直接取用 |

核心取捨：**把「結構化」這件重活，從 query-time 提前到 write-time**。策展當下多付一次成本，把知識寫成結構分明、權威清楚、可溯源的頁面；此後每一次召回都因此又快又準，不再依賴向量相似度的運氣。

---

## 2. 單檔三層架構：拒絕「人一份、AI 一份」

在專案初期，我也曾考慮維護兩份文檔——一份供人類閱讀、一份供 AI 取用。但這種做法的結局如出一轍：兩者逐漸不同步，最終雙方都失去可信度。
因此，我選擇將三種讀者所需的內容，疊合進同一個檔案：

```markdown
---
# 機器層：供 AI 召回與索引使用
features: [手把操作/取消鍵堆疊偵測]
system: UI
authority: tier1-codereview
sources: [raw/codereview/2026-XX-XX.md]
aliases: [手把, 取消鍵, GameOver]
---

## 功能：GameOver 取消鍵改為偵測堆疊頂端 UI 是否仍顯示   ← 閱讀層：供人類閱讀，不夾雜程式碼
取消鍵的處理改為偵測堆疊頂端的 UI 是否確實仍在顯示。[^cancel]

[^cancel]: `Foo/NewUIGameOver.cs:120-140`（示意）。   ← 引用層：人類複查 + AI 溯源
```

- AI 僅需讀取 Frontmatter 即可完成語意召回，無需吃下整頁內容。
- 企劃只需閱讀中段的口語化條列。
- 需追查實際變更的程式碼時，順著 Footnote 直達精確行號。

同一份內容，三種視角，永不分叉。

## 3. 權威仲裁：以「來源可信度」而非「時間」收斂矛盾

我最初的仲裁規則是「以最新日期覆蓋舊資料」——直到它出事：
一張兩個月前的工單規格，竟覆蓋掉了當日剛從真實程式碼提取出的事實。
那次之後，我將仲裁的第一準則改為**來源層級**，而非日期：

```
Tier1 = Code Review（讀取真實程式碼）   ← 最高，代表事實
Tier2 = 工單規格（人為書寫的意圖）       ← 較低，可能尚未實作 / 已變更

同一 Feature 出現多個來源時：
  ├ 存在 Tier1 → 高 Tier 覆蓋低 Tier（忽略日期），被覆蓋者標註「已被實作修正」
  ├ 同 Tier    → 才比對來源日期
  └ Code 與工單矛盾且無 Tier1 可裁決 → 移入 review_queue 等待人工 /resolve
```

效果：知識庫具備**自動去矛盾**的能力，得以將人力集中於少數真正需要判斷的核心衝突。

### 鐵則：LLM 絕不自行推論

仲裁能安全自動化的前提，是**把判斷權留在規則手裡，而非交給 LLM 臆測**。
權威分級（Tier1 > Tier2）、定案 `status`、覆蓋條件，全部由人與規則明定；LLM 只負責「按規則執行」與「把結構化內容寫正確」，**絕不自行推論誰更權威**。
這條鐵則讓整個策展結果可預期、可稽核——萬一出錯，是規則錯，而非模型當天「想法不同」。

## 4. 跨工具整合：被 Windows Symlink 權限坑出來的三層拆解

目標是「一份指令源餵養 Claude Code / Codex / OpenCode 三家」。
我最初採用最直覺的 Symlink，結果安裝腳本在 Windows 上一執行便卡在管理員權限請求——對一個預期由他人安裝的工具而言，這並不可接受。
被此坑所迫，我依「各層所需的機制」將共用部分拆解為三層：

```text
指令檔層：AGENTS.md 作為單一事實來源
          CLAUDE.md 以一行 @AGENTS.md 匯入   （@import 無需提權）

命令層：  .claude/commands  ──junction──▶  .agent/commands
          （Junction 為目錄連接，建立時無需管理員權限；Symlink 才需要）

相容層：  Codex / OpenCode 不展開 @import
          → 由安裝腳本將 spec 內容「內聯」寫入各自的指令檔
```

關鍵知識點：**Junction ≠ Symlink**。Junction 建立目錄連接無需提權，恰好繞過此坑。

## 5. INDEX 防漂移：將「路由表」轉為衍生物

AI 的語意召回仰賴一張 INDEX 路由表。手寫的索引必然腐化，而腐化後的召回會開始默默出錯。
解法是**令 INDEX 無法手寫**——它是各知識頁 Frontmatter 的衍生物：

```bash
# 自所有主題頁的 Frontmatter 重生 INDEX
python scripts/build_index.py

# CI：INDEX 與來源不一致即回傳 exit 1，阻斷 PR
python scripts/build_index.py --check
```

重生時機：頁面變動 / `/curate` 收尾 / 手動觸發 / CI `--check`。
INDEX 永遠等同知識頁的真實狀態，杜絕「文件與索引各說各話」。

## 6. 增量去重：以 content_hash 把關

每日重跑策展時，若未變更的內容也整批重做，既耗時又會污染 diff。
Manifest 記錄每個來源的雜湊值，未變更則略過：

```text
for each raw 來源:
    h = sha(內容)
    if manifest[來源] == h:  略過        # 未變更，不重新策展
    else:                    重新策展 + 更新 manifest[來源] = h
```

策展因此轉為增量模式，僅處理真正發生變動的部分。

## 附：一個「不做」的決策

我原先規劃將企劃端的 Excel 配置自動轉換為 AI 可理解的文檔，並已實際著手實作。
然而**實測資訊轉化率僅約 30%**——產出雜訊偏高，維護成本大於收益。我決定果斷移除整個功能，
僅保留高訊噪比的「程式端」知識。

在 Agent 系統中，**控制進入知識庫的訊噪比**，往往比多接一個來源更為重要。
