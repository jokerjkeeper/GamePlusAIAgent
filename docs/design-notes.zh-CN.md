# 知识引擎设计笔记：非直觉的取舍

> 本文深入支撑整套系统的**知识引擎**，是 README 设计巧思（系统层）的底层展开，并附**示意**程序片段（皆为化名，非真实代码）。
> 本文档的重点在于记录当下的设计判断——**为什么这样选**，而非操作手册。

---

## 1. 为什么不用 RAG：把「结构化」提前到 write-time

最直觉的做法是上 RAG（向量检索）：清洗 → 切 chunk → 嵌入 → 向量库 → 相似度召回。我在前身系统（一套处理长文本的 RAG 管线）上试过，搬到游戏开发数据却严重水土不服，归纳出三个病根：

| 病根 | RAG 的问题 | 本系统的解法 |
|------|-----------|-------------|
| **结构被碾平** | 结构化的规格被切成数百字的 chunk 嵌成一团，问不出「某次改版到底动了哪几条设定」 | LLM 在**策展时**保留结构，写成可读的 wiki 主题页 |
| **定案分不出** | 讨论串切块后，被推翻的中间意见与最终定案语义相近，LLM 分不清谁才是权威 | 知识页带明确 `status`（讨论中 / 定案），定案是被标记的权威记录 |
| **来源 / 决策靠猜** | 「谁决定的、资源在哪」只能靠向量相似度捞，不可靠 | 策展时把来源 / 时间 / 权威写成 Frontmatter 结构字段，查询时直接取用 |

核心取舍：**把「结构化」这件重活，从 query-time 提前到 write-time**。策展当下多付一次成本，把知识写成结构分明、权威清楚、可溯源的页面；此后每一次召回都因此又快又准，不再依赖向量相似度的运气。

---

## 2. 单文件三层架构：拒绝「人一份、AI 一份」

在项目初期，我也曾考虑维护两份文档——一份供人类阅读、一份供 AI 取用。但这种做法的结局如出一辙：两者逐渐不同步，最终双方都失去可信度。
因此，我选择将三种读者所需的内容，叠合进同一个文件：

```markdown
---
# 机器层：供 AI 召回与索引使用
features: [手柄操作/取消键堆叠检测]
system: UI
authority: tier1-codereview
sources: [raw/codereview/2026-XX-XX.md]
aliases: [手柄, 取消键, GameOver]
---

## 功能：GameOver 取消键改为检测堆叠顶端 UI 是否仍显示   ← 阅读层：供人类阅读，不夹杂代码
取消键的处理改为检测堆叠顶端的 UI 是否确实仍在显示。[^cancel]

[^cancel]: `Foo/NewUIGameOver.cs:120-140`（示意）。   ← 引用层：人类复查 + AI 溯源
```

- AI 仅需读取 Frontmatter 即可完成语义召回，无需吃下整页内容。
- 策划只需阅读中段的口语化条列。
- 需追查实际变更的代码时，顺着 Footnote 直达精确行号。

同一份内容，三种视角，永不分叉。

## 3. 权威仲裁：以「来源可信度」而非「时间」收敛矛盾

我最初的仲裁规则是「以最新日期覆盖旧数据」——直到它出事：
一张两个月前的工单规格，竟覆盖掉了当日刚从真实代码提取出的事实。
那次之后，我将仲裁的第一准则改为**来源层级**，而非日期：

```
Tier1 = Code Review（读取真实代码）   ← 最高，代表事实
Tier2 = 工单规格（人为书写的意图）       ← 较低，可能尚未实现 / 已变更

同一 Feature 出现多个来源时：
  ├ 存在 Tier1 → 高 Tier 覆盖低 Tier（忽略日期），被覆盖者标注「已被实现修正」
  ├ 同 Tier    → 才比对来源日期
  └ Code 与工单矛盾且无 Tier1 可裁决 → 移入 review_queue 等待人工 /resolve
```

效果：知识库具备**自动去矛盾**的能力，得以将人力集中于少数真正需要判断的核心冲突。

### 铁则：LLM 绝不自行推论

仲裁能安全自动化的前提，是**把判断权留在规则手里，而非交给 LLM 臆测**。
权威分级（Tier1 > Tier2）、定案 `status`、覆盖条件，全部由人与规则明定；LLM 只负责「按规则执行」与「把结构化内容写正确」，**绝不自行推论谁更权威**。
这条铁则让整个策展结果可预期、可审计——万一出错，是规则错，而非模型当天「想法不同」。

## 4. 跨工具集成：被 Windows Symlink 权限坑出来的三层拆解

目标是「一份指令源喂养 Claude Code / Codex / OpenCode 三家」。
我最初采用最直觉的 Symlink，结果安装脚本在 Windows 上一执行便卡在管理员权限请求——对一个预期由他人安装的工具而言，这并不可接受。
被此坑所迫，我依「各层所需的机制」将共享部分拆解为三层：

```text
指令档层：AGENTS.md 作为单一事实来源
          CLAUDE.md 以一行 @AGENTS.md 导入   （@import 无需提权）

命令层：  .claude/commands  ──junction──▶  .agent/commands
          （Junction 为目录连接，创建时无需管理员权限；Symlink 才需要）

兼容层：  Codex / OpenCode 不展开 @import
          → 由安装脚本将 spec 内容「内联」写入各自的指令档
```

关键知识点：**Junction ≠ Symlink**。Junction 创建目录连接无需提权，恰好绕过此坑。

## 5. INDEX 防漂移：将「路由表」转为衍生物

AI 的语义召回仰赖一张 INDEX 路由表。手写的索引必然腐化，而腐化后的召回会开始默默出错。
解法是**令 INDEX 无法手写**——它是各知识页 Frontmatter 的衍生物：

```bash
# 自所有主题页的 Frontmatter 重生 INDEX
python scripts/build_index.py

# CI：INDEX 与来源不一致即返回 exit 1，阻断 PR
python scripts/build_index.py --check
```

重生时机：页面变动 / `/curate` 收尾 / 手动触发 / CI `--check`。
INDEX 永远等同知识页的真实状态，杜绝「文档与索引各说各话」。

## 6. 增量去重：以 content_hash 把关

每日重跑策展时，若未变更的内容也整批重做，既耗时又会污染 diff。
Manifest 记录每个来源的哈希值，未变更则跳过：

```text
for each raw 来源:
    h = sha(内容)
    if manifest[来源] == h:  跳过        # 未变更，不重新策展
    else:                    重新策展 + 更新 manifest[来源] = h
```

策展因此转为增量模式，仅处理真正发生变动的部分。

## 附：一个「不做」的决策

我原先规划将策划端的 Excel 配置自动转换为 AI 可理解的文档，并已实际着手实现。
然而**实测信息转化率仅约 30%**——产出噪声偏高，维护成本大于收益。我决定彻底移除整个功能，
仅保留高信噪比的「程序端」知识。

在 Agent 系统中，**控制进入知识库的信噪比**，往往比多接一个来源更为重要。
